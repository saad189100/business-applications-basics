Block 0

Setup & the mental model

⏱ 10 min · Goal: everyone running cells, everyone sharing one core idea

First, what even is Python doing?

Your computer only understands raw 1s and 0s. Code you write has to be translated for it — and there are two styles of translator:

  • A compiler translates your whole program up front into machine code, then runs that. Think "translate the entire book, then hand it over." Fast to run, but you wait for the translation and it stops at the first error. (Languages like C, Java work this way.)
  • An interpreter translates and runs your code line by line, on the spot. Think "a live translator speaking as you go." You get results immediately and can test one line at a time.

Python is interpreted. That's exactly why a notebook suits it: you write a little, run it, see the result, and adjust — no "build the whole thing first" step. For learning and for poking at data, that fast feedback loop is a real advantage.

📊 Why a business analyst cares

Interpreted means you can explore data conversationally — load a file, peek, filter, check, tweak — getting an answer at each step instead of writing a full program and hoping. It's closer to how you'd poke around a spreadsheet than to traditional "software engineering."

Colab is a notebook: a page of cells you run one at a time. Three things matter today:

  • Run a cell: click it and press Shift + Enter.
  • Order matters: cells share memory, so a cell can depend on one you ran earlier.
  • When it gets weird: Runtime → Restart clears everything and starts fresh. This fixes most mysteries.

Run your first cell:

print("Hello — I am officially writing Python.")
Hello — I am officially writing Python.
⚠ Watch out

If you run cells out of order, you'll get confusing errors about things "not defined." When in doubt: Runtime → Restart and run all from the top.

❓ Questions & challenges

Why does "Restart and run all" fix so many weird errors?

The notebook keeps a running memory (the "kernel") of every variable you've created. Run cells out of order and that memory can hold old or half-finished values that don't match the code on screen. Restart wipes the memory; "run all" rebuilds it cleanly from the top, so what you see is what you get.

Is Colab "real" Python, or a watered-down version?

It's standard Python running on a Google server — the same language you'd run anywhere. The notebook (cells, the kernel) is just a friendly wrapper around it. Nothing you learn here is Colab-only.

If interpreted is "line by line," isn't Python slow?

For raw number-crunching, yes — Python is slower than a compiled language like C. But it has a clever trick: libraries like pandas are written in fast, pre-compiled code under the hood. So when you call df["Revenue"].sum(), the heavy lifting runs at near-C speed while you still get Python's easy, interactive style. You get the best of both: convenience to write, speed where it counts.

🧠 Think it through

You define x = 10 in cell 3, then delete cell 3 from the page. Can you still use x in cell 4?

Reveal

Yes — until you restart. Deleting a cell removes the text, but the value is already sitting in the kernel's memory. This mismatch (code gone, value still alive) is the single biggest source of "but it worked a second ago" confusion. Restarting is how you force the memory to match the page.

Block 1

Variables & types

⏱ 20 min · Goal: store values, know the basic types, see results
A variable is a labeled box. You put a value in it and use the label later.

The = sign means "store the value on the right in the name on the left."

revenue = 5000
product = "Coffee Mug"
units   = 120
print(product, revenue, units)
Coffee Mug 5000 120

The four everyday types

  • int — whole numbers: units = 120
  • float — decimals: price = 12.99
  • str (string) — text, always in quotes: product = "Coffee Mug"
  • bool — true/false: is_returned = False

Check a type with type():

price = 12.99
print(type(price))
<class 'float'>

Printing a real sentence with f-strings

Put an f before the quotes, then drop variables inside { }:

product = "Coffee Mug"
price = 12.99
print(f"The {product} sells for ${price}.")
The Coffee Mug sells for $12.99.
📊 Business angle

Think of variables as named cells in a spreadsheet. Instead of =B2*C2, you write total = price * units — and the name tells you what it means.

⚠ Watch out — three classic errors

Forgot quotes: product = Coffee Mug → SyntaxError. Text needs quotes.
Typo in a name: using product → NameError: name 'product' is not defined.
Mixing text and numbers: "5" + 5 → TypeError. A number in quotes is text, not a number.

quantity = 7

❓ Questions & challenges

What happens when I do "5" + "5"?

You get "55", not 10. For strings, + means "glue together" (concatenate), not add. Both "5"s are text. To add them as numbers, convert first: int("5") + int("5") → 10.

Why is "5" + 5 an error, but "5" + "5" is fine?

+ only works between two of the same kind: two strings glue, two numbers add. One of each and Python refuses to guess what you meant — that's the TypeError. It would rather complain than silently make the wrong choice.

Can I change the type of a value? (casting)

Yes. int("5") → 5, str(5) → "5", float("3.14") → 3.14. You're not editing the original — you're producing a new value of the new type. This matters constantly because data read from files or typed by a user usually arrives as text.

How do I take input from a user?

input() shows a prompt and returns whatever is typed — always as a string, even if they type digits:

age = input("Your age? ")   # returns text, e.g. "25"
age = int(age)              # convert before doing math
print(age + 1)

In real business work you'll usually read data from a file rather than type it in, but input() is handy while learning.

🧠 Think it through

A quantity arrives from a spreadsheet as the text "120". A teammate writes total = "120" * 3 expecting 360. What do they actually get — and why?

Reveal

They get "120120120". With a string, * 3 means "repeat this text 3 times," not "multiply." Python didn't error because string × integer is a valid operation — just not the one they wanted. The fix: convert first, int("120") * 3. Lesson: an operation running without an error doesn't prove it did the right thing. Always sanity-check the type of data coming in.

Block 2

Operators & operations

⏱ ~15 min · Goal: do arithmetic correctly, and combine yes/no conditions
Operators are the verbs of Python — they take values and produce a new value. Two families today: arithmetic (numbers) and logic (true/false).

Arithmetic operators

print(10 + 3)    # 13   add
print(10 - 3)    # 7    subtract
print(10 * 3)    # 30   multiply
print(10 / 3)    # 3.333...  divide (always gives a decimal)
print(10 ** 2)   # 100  power (10 squared)

Two you may not have met: // and %

// is floor division — divide and throw away the remainder (whole units only). % is modulus — give back only the remainder.

print(17 // 5)   # 3   -> 17 splits into 3 whole 5s
print(17 % 5)    # 2   -> with 2 left over
3 · 2
📊 Business angle

Shipping 17 items in boxes of 5? 17 // 5 = 3 full boxes, 17 % 5 = 2 items left for a partial box. % is also the standard way to test "is this even?" — n % 2 == 0 is True for even numbers.

Order of operations (BODMAS)

Python doesn't read strictly left-to-right. It follows the same precedence you learned in maths — Brackets, Orders (powers), Division/Multiplication, then Addition/Subtraction:

print(10 + 5 * 2)      # 20, not 30  -> the * happens first
print((10 + 5) * 2)    # 30          -> brackets force the + first
print(100 - 20 / 4)    # 95.0        -> the / happens before the -
print(2 ** 3 + 1)      # 9           -> power first, then add
📊 Business angle

This is exactly why a spreadsheet formula like =A1+A2*0.1 can surprise you. When in doubt, add brackets — they cost nothing and make your intent obvious: price + (price * tax_rate).

Comparison operators → True / False

These ask a yes/no question and hand back a boolean:

print(500 > 300)    # True
print(200 == 200)   # True   (two equals = "is equal to?")
print(150 != 150)   # False  ("is not equal to?")
print(10 <= 9)      # False

Logic gates: and / or / not

"Logic gates" are how you combine several true/false conditions into one decision. Python uses plain words:

  • and — true only if both sides are true
  • or — true if at least one side is true
  • not — flips true to false and back
revenue = 800
returned = False

print(revenue > 500 and returned == False)   # True  (both hold)
print(revenue > 1000 or revenue > 500)        # True  (one holds)
print(not returned)                           # True  (flip False)
True · True · True

A quick truth table to keep handy:

ABA and BA or Bnot A
TrueTrueTrueTrueFalse
TrueFalseFalseTrueFalse
FalseTrueFalseTrueTrue
FalseFalseFalseFalseTrue
📊 Business angle

Real rules are layered: "flag the order if it's over $500 and not already returned." Each condition is a comparison; and/or stitch them into one test you can drop straight into an if (next block but one).

❓ Questions & challenges

When do I use / vs // ?

/ always gives a decimal (10 / 4 → 2.5), even when it divides evenly (8 / 2 → 4.0). // gives the whole-number part only (10 // 4 → 2). Use // when a fraction makes no sense — number of full boxes, complete months, whole tables of guests.

What's the real-world use of modulus (%)?

The two big ones: testing divisibility (n % 2 == 0 → even; n % 3 == 0 → every third) and "wrapping around" (hour % 12, day-of-week cycles). In business data it's handy for "every Nth row," splitting into groups, or pulling cents from a total in pence.

Is and/or the same as & / | I've seen elsewhere?

For plain true/false logic in Python, use the words and / or / not — they read clearly and are correct. The symbols & | ~ do a lower-level, value-by-value version. You'll meet them again in pandas, where filtering a whole column requires & and | (with each condition in brackets) — but for everyday if statements, stick to the words.

🧠 Think it through

Without running it, what does each line give — and why are they different?

print(2 + 3 * 4 ** 2)
print((2 + 3) * 4 ** 2)
print(((2 + 3) * 4) ** 2)
Reveal

Line 1 = 50: power first (4**2 = 16), then 3*16 = 48, then +2. Line 2 = 80: brackets force 2+3 = 5, but the power still beats the multiply, so 5 * 16. Line 3 = 400: the inner brackets make 5 * 4 = 20 happen first, then square it. Same four numbers, three answers — precedence and brackets decide everything. The business lesson: never trust a chained formula at a glance; bracket it so a colleague reads it the way you meant it.

Block 3

Lists & dictionaries

⏱ 20 min · Goal: hold many values, and look things up by name
A list is a column of values in order. A dictionary is a labeled lookup.

Lists — an ordered column

sales = [200, 450, 130, 600]
print(sales[0])      # first item — counting starts at 0
print(len(sales))    # how many items
print(sum(sales))    # add them all up
sales.append(310)    # add a new value to the end
print(sales)
200 · 4 · 1380 · [200, 450, 130, 600, 310]
📊 Business angle

A list is one column of your spreadsheet — say, daily sales. sum() is your column total, len() is the row count.

Dictionaries — look up a value by its name

prices = {"mug": 12, "shirt": 25, "sticker": 3}
print(prices["shirt"])     # look up by key
prices["hat"] = 18         # add a new entry
print(prices)
25 · {'mug': 12, 'shirt': 25, 'sticker': 3, 'hat': 18}

Which do I use? A list when order/sequence matters (sales over time). A dictionary when you want to look something up by a label (price of a product).

⚠ Watch out

The first item is sales[0], not sales[1]. Asking for sales[4] on a 4-item list → IndexError. Asking a dict for a key that isn't there → KeyError.

❓ Questions & challenges

Where did len(), sum(), and append() come from? I never defined them.

They ship with Python — you didn't write them, but they're always available. len() and sum() are built-in functions. append() is a little different: it's a method that belongs to lists, which is why you call it on a list with a dot — sales.append(5) — rather than append(sales, 5).

Function vs method — what's the actual difference?

A function stands on its own: len(sales). A method is attached to an object and called with a dot: sales.append(5). Same core idea (inputs → action or result), different grammar. A method can see and change the thing it's attached to, which is why sales.append(5) modifies sales directly. Rule of thumb: name(stuff) = function; thing.name(stuff) = method.

Why don't len() and sum() work on a single value, like sum(5)?

Because they're designed to work over a collection. sum needs several things to add together; len needs items to count. A lone 5 is not a collection — there's nothing to iterate over — so you get a TypeError. Wrap it in a list and it works: sum([5]) → 5.

What's the difference between a list and a tuple?

A list uses [ ] and can be changed — add, remove, edit. A tuple uses ( ) and is locked once created: you can read it but not change it. Use a tuple for things that shouldn't move (a fixed pair like (latitude, longitude) or a row that represents one record); use a list for collections you intend to modify.

What does sales[1:3] mean? (slicing)

A slice grabs a range of items, list[start:stop] — and stop is not included:

sales = [200, 450, 130, 600, 310]
sales[1:3]   # [450, 130]  -> positions 1 and 2, not 3
sales[:2]    # [200, 450]  -> from the start
sales[2:]    # [130, 600, 310]  -> to the end
sales[-1]    # 310  -> the last item
What can and can't I put inside the square brackets?

It depends on what's in front of them. After a list, brackets mean "which position(s)": a number (sales[0]), a slice (sales[1:3]), or a negative index (sales[-1]). After a dictionary, brackets mean "which key": prices["mug"]. Put a key into a list, or a position into a dict, and you get IndexError / KeyError. You can't put a calculation that produces text where a position is expected.

Can I do prices[1, "shirt"] — two things in the brackets?

Not with a plain list or dictionary — they expect a single position or key, so that comma is meaningless to them. But pandas (the data-table library, Block 5) does use it: df.loc[1, "Revenue"] means "row 1, column Revenue." The comma separates two axes — rows, then columns. That's a pandas feature, not core Python.

How does range() work?

It produces a sequence of numbers, again with the end excluded:

range(5)        # 0, 1, 2, 3, 4
range(2, 6)     # 2, 3, 4, 5
range(0, 10, 2) # 0, 2, 4, 6, 8  (step of 2)

It's mostly used to repeat something a set number of times: for i in range(5):. We'll use it in the next block.

🧠 Think it through

Two-parter. (a) What does sales[2:2] return? (b) Your colleague stores a product record as record = ("Mug", 12) then tries record[1] = 15 to update the price. What happens, and what should they have used instead?

Reveal

(a) An empty list [] — when start equals stop there's nothing between them. (b) record[1] = 15 raises a TypeError: a tuple is locked, you can't reassign its items. If the price needs to change, the record should have been a list ["Mug", 12] — or, better for real data, a dictionary {"product": "Mug", "price": 12} so the fields have names. The deeper point: choosing list vs tuple vs dict is a design decision about whether the data should change and how you'll look it up.

Block 4 · core

Logic & loops

⏱ 25 min · Goal: make decisions and repeat work — what makes code more than a calculator

Comparisons

These ask a yes/no question and give back True or False:

print(500 > 300)     # greater than
print(200 == 200)    # equal? (two equals signs!)
print(150 != 150)    # not equal?
True · True · False
⚠ Watch out — the #1 beginner trap

= stores a value. == compares two values. x = 5 puts 5 in x; x == 5 asks "is x equal to 5?"

Making decisions with if / elif / else

amount = 420

if amount >= 500:
    print("Large order")
elif amount >= 200:
    print("Medium order")
else:
    print("Small order")
Medium order
⚠ Watch out — indentation is not decoration

Python uses the spaces at the start of a line to know what's "inside" the if. Forget them and you get IndentationError. Colab indents for you after the colon : — let it.

Repeating work with a for loop

A loop runs the same code once for each item in a list:

orders = [150, 620, 90, 540, 300]

large_count = 0
for amount in orders:
    if amount >= 500:
        large_count = large_count + 1

print(f"Large orders: {large_count}")
Large orders: 2
📊 Business angle

A loop is "go down the column and do this to every row." Counting large orders, flagging late shipments, summing a column — all the same pattern.

The other loop: while

A for loop runs once per item in a known collection. A while loop runs as long as a condition stays true — you don't say how many times up front:

balance = 1000
months = 0
while balance > 0:        # keep going until the money runs out
    balance = balance - 300
    months = months + 1
print(f"Funds last {months} months")
Funds last 4 months

When is each more suitable?

Use a for loop when…Use a while loop when…
You're walking through a known list/table (every order, every row)You repeat until something happens, and don't know the count ahead of time
The number of repeats is fixed (range(12) for 12 months)"Keep drawing down the budget until it hits zero"
99% of business data work — you have data to iterate overSimulations, retrying until success, "grow until target reached"
⚠ Watch out — the infinite loop

A while loop only stops when its condition becomes false, so something inside must move it toward false. Forget to change balance and it loops forever. If a cell hangs, press the stop button or Runtime → Interrupt.

❓ Questions & challenges

Why two equals signs (==) instead of one?

One = stores a value (x = 5 puts 5 into x). Two == ask a question (x == 5 means "is x equal to 5?"). Keeping them separate stops "set this" and "compare this" from being confused — but writing = where you meant == is a classic bug.

How do I repeat something a fixed number of times instead of walking a list?

Loop over range():

for i in range(3):
    print("Reminder", i)   # i is 0, then 1, then 2

Use for x in sales: when you want each item; use range() when you just want to repeat by count.

Can I loop over a dictionary?

Yes. for key in prices: gives you each key, and prices[key] gets its value. To get both at once:

for product, price in prices.items():
    print(product, "costs", price)
🧠 Think it through

This loop is supposed to count orders over $500, but it always prints 1. Spot the bug before reading on.

orders = [620, 150, 540, 900]
for amount in orders:
    large_count = 0
    if amount > 500:
        large_count = large_count + 1
print(large_count)
Reveal

large_count = 0 sits inside the loop, so it resets to zero on every pass — the count can never climb past 1. The counter must be created once, before the loop, so it survives across iterations. This is a pure indentation/placement bug, and reasoning about "what runs how many times" is exactly the skill that separates copying code from understanding it.

Block 5

Functions

⏱ 15 min · Goal: reuse logic, and understand the function calls you already see everywhere
A function is a named machine: inputs go in, a result comes out.

You've been using functions all along — print(), len(), sum(), round(), max(), min() are all built in:

sales = [200, 450, 130, 600]
print(max(sales))      # biggest
print(min(sales))      # smallest
print(round(sum(sales) / len(sales), 1))   # average, 1 decimal
600 · 130 · 345.0

Write your own with def

def with_tax(price, rate=0.1):
    return price * (1 + rate)

print(with_tax(100))        # uses default 10%
print(with_tax(100, 0.2))   # override to 20%
110.0 · 120.0

Why bother: name a calculation once and reuse it everywhere. If the rule changes, you fix it in one place instead of twenty.

⚠ Watch out — print vs return

print shows something on screen for a human. return hands a value back so your code can keep using it. A function that only prints can't be used in a calculation later.

❓ Questions & challenges

When I write df.head(), is that a function or a method?

A method — it belongs to the DataFrame df and is called with a dot. print() is a function; it stands alone. Most pandas operations are methods because they act on one specific table. (Same distinction as sales.append() back in Block 2.)

What does rate=0.1 in the definition mean?

It's a default value. If the caller doesn't supply rate, it falls back to 0.1. That's why with_tax(100) works with no second value, while with_tax(100, 0.2) overrides it.

Argument vs parameter — are they the same thing?

Almost, and people use them loosely, but: in def with_tax(price, rate): the names price and rate are parameters (the placeholders). In the call with_tax(100, 0.2) the values 100 and 0.2 are arguments (the real inputs). Knowing this makes error messages like "missing 1 required positional argument" readable.

What if my function has no return?

It hands back None — Python's word for "nothing." A function that only prints returns None, so x = my_function() leaves x empty. If you want to use the result later, you must return it.

🧠 Think it through

What does this print, and why?

def add(a, b):
    a + b

result = add(2, 3)
print(result)
Reveal

It prints None. The function computes a + b but never returns it, so the result is calculated and immediately thrown away. add(2, 3) hands back None, and that's what lands in result. Forgetting return — and confusing it with print — is one of the most common beginner bugs. Fix: return a + b.

display() vs print() — why your table sometimes looks ugly

In Colab, whatever sits on the last line of a cell is shown automatically as a nicely formatted table. That's display() working behind the scenes. print() instead dumps a plain-text version — fine for numbers and words, but cramped for DataFrames.

df.head()          # ✅ rich, aligned table (auto-displayed)
display(df.head()) # ✅ same thing, and works mid-cell / in loops
print(df.head())   # ⚠ plain monospaced text — harder to read
  • Showing one table? Just put it on the last line.
  • Showing several in one cell, or one from inside a loop/if? Use display(...) — only the last line auto-shows, so the others would otherwise be invisible.
  • Use print() for plain values and sentences (print(f"Total: {total}")), not for tables.

df["Profit"] = df["Revenue"] - df["Cost"]
df["Margin"] = df["Profit"] / df["Revenue"]

Find and handle missing data (real datasets always have gaps):

df.isna().sum()          # how many blanks in each column
df["Cost"].fillna(0)     # replace blanks with 0
df.dropna()              # drop any row that has a blank

❓ Questions & challenges

I wrote condition = df["Revenue" > 500]. Why does it break?

Look at where the > sits — it's inside the brackets. So Python first evaluates "Revenue" > 500, i.e. compares the word "Revenue" to the number 500, before pandas ever sees it. The brackets should hold the column, and the comparison goes outside:

condition = df["Revenue"] > 500   # column first, then compare
big = df[condition]

The one-liner df[df["Revenue"] > 500] is exactly the same thing, just without naming the middle step.

So what actually is "condition"?

A column of True/False — one value per row, often called a mask. df[condition] keeps only the rows where it's True. It's Block 3's comparison, applied to every row at once instead of one value at a time.

Why does df["ages"] > 30 work, but a plain list ages > 30 doesn't?

A plain Python list is just a generic container. ages > 30 asks Python to compare the whole list object to the number 30 — and Python has no built-in meaning for "a list versus a number," so it raises a TypeError. The list won't look inside itself for you; you'd have to loop:

ages = [25, 40, 30, 55]
[a > 30 for a in ages]      # [False, True, False, True]

A pandas column is a different kind of object — a Series — and its creators deliberately defined what > 30 means: "apply this to every element and hand back a new Series of True/False." That behaviour is called vectorization. It's the same reason df["price"] * 1.1 bumps every price by 10%, while [10, 20] * 1.1 would error.

Rule of thumb: a plain list needs a loop to act on every element; a pandas Series does it for you in one go. That auto-broadcasting is a big part of why pandas feels so much faster to write than raw Python.

I filtered for revenue over 500 — how do I get just the "Returns" column for those rows?

Filter the rows, then pick the column:

big = df[df["Revenue"] > 500]
big["Returns"]

Or do both at once with .loc, which takes [rows, columns]:

df.loc[df["Revenue"] > 500, "Returns"]
That comma in .loc[rows, columns] — is it the prices[1, "shirt"] from Block 2?

Yes — this is where that idea pays off. pandas reads the brackets as two axes: rows first, then columns. df.loc[2, "Revenue"] is "the Revenue cell in row 2." Plain lists and dicts can't take two things in their brackets; pandas can, because a table has two dimensions.

What's the difference between df["Revenue"] and df[["Revenue", "Cost"]]?

Single brackets with one name give you one column (a Series). Double brackets — brackets around a list of names — give you a smaller table with just those columns. The inner [ ] is literally a list of column names, the same kind of list from Block 2.

How do I filter on two conditions at once? (and the & / | surprise)

Here's the catch from the operations block: when filtering a whole column, pandas needs the symbols & (and) and | (or), not the words — and each condition must be in its own brackets:

# big orders in the North region
df[(df["Revenue"] > 500) & (df["Region"] == "North")]

Forget the inner brackets and you'll get a confusing error, because & binds tighter than >. This is the one place the everyday and/or rule flips.

.loc vs .iloc — what's the difference?

.loc selects by label (column names, the row's index value): df.loc[0, "Revenue"]. .iloc selects by integer position, like a list: df.iloc[0, 2] = first row, third column. Use .loc when you know the names, .iloc when you just want "the first 5 rows" — df.iloc[:5].

Errors & where to go next

⏱ 5 min · Goal: leave unafraid of red text

Errors aren't failure — they're Python telling you exactly what and where. Read the bottom line first: it names the error type and the line number.

The four errors you'll meet 90% of the time

ErrorWhat it usually meansThe fix
NameErrorUsed a variable that doesn't exist (often a typo)Check spelling; run the cell that defines it
TypeErrorMixed incompatible types, e.g. text + numberConvert with int()/str()
IndexError / KeyErrorAsked for an item/key that isn't thereRemember counting starts at 0; check the key exists
IndentationErrorWrong spacing inside an if/for/defLine up the indented block consistently

Your debugging habit

  • Print the variable to see what's actually inside it.
  • Re-run from the top — Runtime → Restart and run all.
  • Ask for help well: paste the code + the full error message. That's 90% of getting a useful answer from docs, a classmate, or an AI assistant.

Reference docs: Python docs · pandas docs

❓ Questions & challenges

Is reading just the last line of an error really enough?

About 80% of the time, yes — the last line names the error type and the message, and the line number points you close. The block above it (the "traceback") shows the path the code took to get there, which matters when the error happens deep inside a function or a library you called.

How do I ask an AI or a classmate for help well?

Give them three things: the code that broke, the full error message (not a paraphrase), and what you expected to happen. That trio answers 90% of follow-up questions before they're asked — and forcing yourself to write it out often reveals the bug on its own.

🧠 Think it through

Python says the error is on line 12, but line 12 looks perfectly fine. Where else should you look first?

Reveal

The line above — line 11. An unclosed bracket, quote, or parenthesis on one line makes Python keep reading onto the next, so it only notices something is wrong when it hits line 12 and blames it. Whenever the flagged line looks innocent, check the line before for a missing ), ], or quote. Reported line numbers point to where Python got confused, not always where you made the mistake.
